# 入门基础笔记

---

## Day 1 — 协议入门 + 发布订阅模型

### 1. MQTT 是什么

MQTT（Message Queuing Telemetry Transport）是一种基于**发布-订阅模式**的轻量级消息传输协议，专为资源受限设备和低带宽、高延迟网络设计，广泛用于物联网（IoT）场景。

**协议版本：**
- MQTT 3.1.1：当前应用最广泛的版本
- MQTT 5.0：新增用户属性、原因码、共享订阅等特性（后续重点学习）

**标准端口：**
- `1883`：TCP 明文传输
- `8883`：TLS/SSL 加密传输

---

### 2. 核心组件

#### MQTT Broker（代理/服务器）
负责所有消息的**接收、路由和转发**，同时处理客户端的连接、断开、订阅、取消订阅请求。常见实现：Mosquitto、EMQX、HiveMQ。

#### Publisher（发布者）
向特定 Topic 发送消息的客户端。发布时**无需关心是否有订阅者**，也无需与订阅者建立直接连接。一次只能向一个 Topic 发送数据。

#### Subscriber（订阅者）
向 Broker 订阅 Topic 来接收消息，支持**一次订阅多个主题**（含通配符）。

#### Topic（主题）
消息路由的基础，本质是一个 **UTF-8 编码字符串**，用 `/` 分层。Broker 据此将消息匹配转发给对应订阅者。

---

### 3. 发布/订阅模式的三维解耦

这是 pub/sub 模式相比传统请求/响应模式最核心的优势：

| 解耦维度 | 含义 |
|---------|------|
| **空间解耦** | 发布者和订阅者互不知晓对方的存在（IP、端口均不暴露） |
| **时间解耦** | 发布者和订阅者不需要同时在线（配合持久会话可缓存离线消息） |
| **同步解耦** | 发布和订阅操作都是异步的，不阻塞双方 |

> 副作用：发布者无法直接知道订阅者是否收到消息。MQTT 5.0 为此引入了**请求/响应模式**作为补充。

---

### 4. MQTT 报文基本结构

每个 MQTT 控制报文由最多三部分组成：

```
┌─────────────────┐
│   Fixed Header  │  ← 所有报文都有，1~5 字节
│  (固定报头)      │     包含：报文类型(4bit) + 标志位(4bit) + 剩余长度
├─────────────────┤
│ Variable Header │  ← 部分报文有，内容因报文类型而异
│  (可变报头)      │     如 CONNECT 包含协议名、协议级别、连接标志、Keep Alive
├─────────────────┤
│    Payload      │  ← 部分报文有，实际消息内容
│   (有效载荷)     │     如 CONNECT 的 Client ID、Will Topic、用户名密码
└─────────────────┘
```

**报文类型（部分）：**

| 类型 | 名称 | 方向 |
|------|------|------|
| CONNECT | 连接请求 | 客户端 → 服务器 |
| CONNACK | 连接确认 | 服务器 → 客户端 |
| PUBLISH | 发布消息 | 双向 |
| SUBSCRIBE | 订阅请求 | 客户端 → 服务器 |
| PINGREQ/PINGRESP | 心跳请求/响应 | 双向 |
| DISCONNECT | 断开连接 | 客户端 → 服务器 |

---

### 5. 工作流程

```
1. 建立连接
   客户端发送 CONNECT 报文（含 ClientID、Clean Session、Keep Alive、可选用户名密码）
   Broker 回复 CONNACK 确认

2. 发布/订阅
   发布者：向指定 Topic 发送 PUBLISH 报文
   订阅者：发送 SUBSCRIBE 报文声明感兴趣的 Topic

3. 消息路由
   Broker 根据 Topic 将消息转发给所有匹配的订阅者
   按 QoS 等级保证传递可靠性

4. 断开连接
   正常断开：客户端发送 DISCONNECT 报文
   异常断开：Broker 检测到超时，触发 LWT（遗嘱消息）
```

> **ClientID**：每个客户端连接 Broker 时必须提供唯一标识符，Broker 用它来识别持久会话。

---

### 6. 主题（Topic）规则与通配符

#### 命名规则
- 大小写敏感（`sensor/temp` ≠ `sensor/Temp`）
- 不建议以 `/` 开头或结尾（如 `/chat` 或 `chat/`）
- 不能包含空字符（`\u0000`）

#### 单层通配符 `+`
匹配某一层级的任意内容，必须单独占据整个层级：

```
sensor/+/temperature   ✓  → 匹配 sensor/1/temperature, sensor/2/temperature
sensor+                ✗  → 无效（没有占据整个层级）
```

#### 多层通配符 `#`
匹配任意数量的子层级，必须是主题的最后一个字符：

```
sensor/#               ✓  → 匹配 sensor, sensor/temp, sensor/1/temp/unit
sensor/#/temperature   ✗  → 无效（不是最后一个字符）
sensor/bedroom#        ✗  → 无效（没有占据整个层级）
```

> **注意**：通配符只能用于**订阅**，不能用于发布。

#### 特殊主题
- `$SYS/` 开头：**系统主题**，用于获取 Broker 运行状态、连接数、消息统计等（各 Broker 实现不同）
- `$share/` 开头：**共享订阅主题**（MQTT 5.0 规范），用于多订阅者负载均衡

### 7. QoS 服务质量等级

| 等级 | 语义 | 机制 |
|------|------|------|
| QoS 0 | 最多一次（At most once） | 如果当时客户端不可用，则会丢失该消息。发布者发送一条消息之后，就不再关心它有没有发送到对方，也不设置任何重发机制 |
| QoS 1 | 至少一次（At least once） | 包含了简单的重发机制，发布者发送消息之后等待接收者的 ACK，如果没收到 ACK 则重新发送消息。这种模式能保证消息至少能到达一次，但无法保证消息重复 |
| QoS 2 | 恰好一次（Exactly once） | 四步握手（PUBLISH→PUBREC→PUBREL→PUBCOMP）设计了重发和重复消息发现机制，保证消息到达对方并且严格只到达一次 |

> Day 3 将深入学习 QoS 机制和会话持久化，这里先建立基本印象。

### 8. 三个特殊机制（入门了解）

#### 保留消息（Retained Message）
发布时设置 `retain=true`，Broker 会保存该消息。**后续任何新订阅者订阅该 Topic，会立即收到这条最新保留消息**（不必等待下次发布）。常用于设备状态同步。

#### 遗嘱消息（Last Will and Testament, LWT）
客户端在 CONNECT 时预设遗嘱消息（Topic + Payload）。当客户端**异常断开**（未发送 DISCONNECT）时，Broker 代为发布遗嘱消息。常用于设备离线检测。

#### Clean Session（清洁会话）
- `true`：断开后 Broker 清除所有会话状态（订阅关系、未送达消息）
- `false`：断开后 Broker 保留会话，重连后继续接收离线期间的消息（需相同 ClientID）

> MQTT 5.0 将此拆分为 **Clean Start** 和 **Session Expiry Interval** 两个字段，更灵活。

### 9. 协议对比速查

| 对比维度 | MQTT | HTTP | WebSocket |
|---------|------|------|-----------|
| 通信模式 | 发布/订阅（异步） | 请求/响应（同步） | 全双工（持久连接） |
| 报文开销 | 极小（最小 2 字节） | 较大（HTTP 头部） | 中等 |
| 适用场景 | 大量设备、低带宽、实时遥测 | 简单上报、主动拉取 | 浏览器实时交互 |
| 离线消息 | 支持（持久会话） | 不支持 | 不支持 |

**MQTT over WebSocket**：将 MQTT 报文封装在 WebSocket 帧中，使浏览器可以直接接入 MQTT 网络。

**MQTT over QUIC**（EMQX 5.0+）：基于 UDP，适合弱网和频繁切换网络的场景（如车联网、移动设备）。

### 10. Day 1 小结

```
核心模型：Client(Pub/Sub) ←TCP→ Broker ←TCP→ Client(Pub/Sub)
路由依据：Topic（UTF-8字符串，/分层，支持+/#通配符）
可靠保证：QoS 0/1/2
特殊机制：Retained Message + LWT + Clean Session
协议优势：三维解耦（空间/时间/同步）+ 轻量报文头
```
